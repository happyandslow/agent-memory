# Agent-Memory Workflow Redesign — Design Spec

Date: 2026-07-05
Owner: Le Xu
Status: Approved design; implementation planning next.

## Problem

Work-project context is maintained across three actors — the human (Le), Claude
Code (coding, mostly remote on SSH machines), and Hermes (local logistics/backstop)
— sharing one durable memory layer: the `agent-memory` git repo, whose
`projects/<project>` folders are symlinked into the Obsidian vault under `10-work/`.

The system is well-designed on paper but has drifted in practice:

1. **Triplicated state.** "Current focus", "next actions", and the key-result
   summary are hand-authored simultaneously in `memory/context.md`,
   `tracking/status.md`, and `index.md` (confirmed in WaferEngine and nc_service).
   Updating one drifts the others.
2. **Declutter debt.** Hermes's daily maintenance cron appends no-op lines
   (`"Daily maintenance found no new project notes…"`) to `status.md`, growing noise
   instead of curating.
3. **Allocated-but-empty scaffolding.** Every project copies the full `_template`
   skeleton (`topics/`, `transcripts/`, `inbox/`, `agents/`, `docs/`, `human/`), most
   holding only placeholder `README.md`. `WaferEngine-staging` is an empty duplicate
   of `WaferEngine`.
4. **No single home for the maintenance procedure.** The logic lives in THREE places
   that drift: Hermes's cron inline prompt, the Hermes `synced-project-memory-vault`
   skill + its 6 reference docs, and the repo's `AGENTS.md`.
5. **Remote Claude writes off-spec.** When Claude Code works remotely it dumps
   durable notes into `memory/topics/` with no date indexing, so chronology is
   untrackable and topic notes are neither evergreen nor few.
6. **Scattering across stores.** Some deep context lives in external ContextBase docs
   (e.g. nc_service `GOZQ9I8pOe`) in addition to git.

Underlying tension: the human wants **few files, append-only**; agents want
**small, sliced, indexed files** so no single file grows huge and not all context
must load at once. Today the system delivers the worst of both — big files AND
scattered duplicate copies.

## Goals

- Each fact has exactly one authored home; duplicate surfaces become generated projections.
- The human maintains only two files routinely (a braindump + a roadmap).
- Any agent writing to `agent-memory` follows a standard format and placement (a "write contract").
- Chronology is always trackable via a generated date index.
- The maintenance procedure has one versioned home that both Claude Code and Hermes
  invoke, on both Mac and remote machines, updated via `git pull`.
- The maintenance pass is safe-autonomous: auto-commit mechanical curation, never
  auto-resolve ambiguous human conflicts.
- The skill is itself improvable over time.

## Non-goals

- Migrating ContextBase content into git. ContextBase remains the external deep-log
  store; the write contract only requires a single pointer to it from `project.md`.
- Changing how the Obsidian symlinks work (they already collapse vault ↔ repo to one surface).
- Building deterministic-only tooling; judgment steps stay agent-driven.

## Design

### 1. Target per-project file model

Only two files the human touches routinely. Everything else is rarely-touched-stable
or fully generated.

**A. Human-authored (source of truth)**

| File | Role | Human touches |
|---|---|---|
| `capture.md` (new) | Single append-only human braindump: todos, meeting scraps, any format. | Constantly |
| `plan.md` | Roadmap: goals, milestones, decisions, and the canonical next-actions checklist. | Occasionally |
| `memory/project.md` | Stable facts: repo paths, machines, commands, conventions, single ContextBase pointer. | Rarely |
| `memory/topics/<slug>.md` | Evergreen deep knowledge, one per subsystem, stable name, updated in place. The hierarchy that keeps files small. | When knowledge deepens |

**B. Agent-written dated captures + human raw dumps**

| Path | Role |
|---|---|
| `memory/inbox/YYYY-MM-DD-<topic>.md` | Dated, time-bound findings / session outcomes from Claude (and any agent). Standard format. Drained by maintenance. |
| `meetings/` (new) | Shared working area for meeting slides/materials (pptx, pdf, images), co-edited by the human and agents. Placed outside `docs/` and exempt from the dated-filename checker: **names are freeform** (dates live in slide content; chronology tracked via git history/mtime/in-slide dates). Agents may create/append/modify slides here on request. Maintenance reads it to distill decisions/TODOs into `capture.md`/`plan.md`/`timeline.md` but never reorganizes the folder on its own. |

**C. Generated views** — each carries a header `<!-- GENERATED by agent-memory skill: do not hand-edit -->` and is rebuilt from A + B.

| File | Role |
|---|---|
| `memory/context.md` | Compact session-start packet: a thin router — current focus + pointers to must-read topics + next actions. Never inlines what lives elsewhere. |
| `tracking/status.md` | Dashboard: current state, next actions, risks, freshness. |
| `memory/timeline.md` (new) | The date index. One line per dated event → pointer. Built from dated inbox files, session/transcript pointers, and git log. |
| `index.md` | Obsidian landing page — links only. |

Because A/B never repeat what C renders, the triplication is eliminated by
construction: next-actions are authored once in `plan.md`; `context.md` and
`status.md` are pure projections.

### 2. Write contract (fixes the topics/ dump)

Any agent writing to `agent-memory` follows these placement rules. The governing
principle: **during a live session an agent captures to dated `inbox/` by default;
`topics/` consolidation happens later in the maintain pass.** This is what keeps
`topics/` few and clean — new topic files only ever come from curation.

- **Default write during a session** → a new `memory/inbox/YYYY-MM-DD-<topic>.md` in the
  standard capture template. Maintenance later distills it into the right topic and
  records a line in `timeline.md`.
- **Allowed exception** → an agent may *append to an existing* `memory/topics/<slug>.md`
  **in place** (via a dated `## Updates` subsection) when it is clearly refining that
  topic's evergreen facts. It may **not** create new topic files — that is the maintain
  pass's job.
- **Maintain pass owns `topics/`** → it distills inbox entries into existing topics and
  creates new topic notes as evergreen knowledge accumulates.
- **A generated view** is never hand-written by a session; it is produced by the skill.
- **Meeting slides** (`meetings/`) are co-edited artifacts. An agent asked to
  create/append/modify a deck: pulls latest first, edits the one working file in place
  (no proliferating copies), commits promptly, and — if it cannot tell whether the human
  has unsaved local edits — asks rather than overwrites.
- Rule of thumb: *"Evergreen → the topic note. Time-stamped → a dated inbox file.
  Never a new bare file in topics/. Never hand-edit a generated view."*

Standard templates ship with the skill (`templates/`): capture entry, topic note,
session/transcript pointer, dated docs artifact. Each defines a title/date/frontmatter/
section layout so the maintenance pass can parse and merge reliably.

Enforcement is two-layered: (1) the work-repo routing snippet + Claude-side skill tell
a fresh remote session the format/placement before it invents its own; (2) the
maintenance pass fixes drift — renames undated files, re-routes misplaced topic dumps,
rebuilds indexes.

### 3. Skill architecture — one shared brain, two thin wrappers

The canonical procedure has one home. Because Claude Code and Hermes load different
skill frontmatter formats, the shared artifact is a harness-agnostic **`PLAYBOOK.md`**,
wrapped by two thin adapters that both say "follow PLAYBOOK.md":

- **Claude Code adapter** — the `SKILL.md` in the skill package (native).
- **Hermes adapter** — `synced-project-memory-vault` slims to *"read
  `<claude-skills>/agent-memory/PLAYBOOK.md` and follow it"*; the cron's giant inline
  prompt collapses to the same one line.

Edit `PLAYBOOK.md` once → `git pull` → both agents and both machines update.

**Two operations:**

| Op | Who / when | What |
|---|---|---|
| write | either agent, during/after a session | Enforce the write contract: right file, right format, compact. Cheap. |
| maintain | Hermes cron daily + either agent on demand | The heavy curation pass (below). |

**The `maintain` pass, in order:**

1. **Deterministic pre-flight** (scripts): structure check; list undated files, empty
   scaffold dirs, cross-file duplication candidates, un-drained captures.
2. **Drain** `capture.md` + dated `inbox/` → distill into `plan.md` next-actions /
   `topics/` / decisions; mark each drained item. Also scan `meetings/` for new slides
   and distill any decisions/TODOs into `capture.md`/`plan.md`/`timeline.md` (read-only;
   never move/rename/delete the slide files).
3. **Regenerate views** `context.md`, `status.md`, `index.md`, `timeline.md` from
   sources (deterministic where possible; agent-written prose where needed).
4. **Declutter**: dedupe; prune empty dirs; append a `status.md` "Recent changes" line
   **only when something actually changed** (no-op runs bump a single freshness
   timestamp).
5. **Conflicts**: ambiguous/contradictory items → `tracking/conflicts.md` + surfaced to
   the human. Never auto-resolve human conflicts.
6. **Commit** safe changes; report.

**Autonomy:** auto-commit mechanical curation (drain, regenerate, dedupe, rename,
prune); flag ambiguous items to `conflicts.md`.

**Self-improvement:** `PLAYBOOK.md` is versioned. When `maintain` hits an uncovered
case, it drops a dated note in `references/` and, when warranted, proposes a
`PLAYBOOK.md` edit — the reference-doc pattern Hermes already uses, now shared.

### 4. Packaging & install

The skill is a self-contained package added to `github.com/happyandslow/claude-skills`
(default branch `master`, flat layout of self-contained skills):

```
claude-skills/agent-memory/
  SKILL.md            # Claude-format frontmatter; body delegates to PLAYBOOK.md
  PLAYBOOK.md         # the single source of truth procedure
  write-contract.md   # placement + format rules for any agent
  templates/          # capture entry, topic note, session pointer, docs artifact
  references/         # dated case notes (migrated from Hermes's 6 docs)
  scripts/            # build_views.py, find_conflicts.py, check_memory_repo.py (take --root)
  install.sh          # one-command bootstrap
```

Scripts take `--root $AGENT_MEMORY_ROOT` so the same code runs on Mac and gala2
against the local agent-memory checkout.

**Install = bootstrap script + symlinks** (update via `git pull`):

| Surface | Command | Effect |
|---|---|---|
| Mac Claude Code | `~/Projects/claude-skills/agent-memory/install.sh` | symlink → `~/.claude/skills/agent-memory` |
| Remote gala2 Claude Code | clone repo, run same `install.sh` | symlink → `~/.claude/skills/agent-memory`; set `AGENT_MEMORY_ROOT=~/agent-memory` |
| Hermes | `install.sh --hermes` | writes slim wrapper to `~/.hermes/skills/note-taking/synced-project-memory-vault` |
| Each work repo | routing snippet update (via `init_project.py`) | fresh remote sessions load the write-contract before writing |

"Instruct everyone" reduces to: **clone `claude-skills`, run
`agent-memory/install.sh` (add `--hermes` on the Hermes host)."**

### 5. One-time cleanup / migration

**In `agent-memory`:**

1. Prune empty-scaffold template: `_template` stops shipping empty placeholder-README
   dirs; dirs are created lazily on first real file. `init_project.py` scaffolds only
   `plan.md`, `project.md`, `capture.md`, the generated views, and the human-input
   `meetings/` dump dir (kept intentionally, not auto-scaffold noise).
2. Migrate the 4 existing projects to the new model: add `capture.md`, generate
   `timeline.md`, de-duplicate `context.md`/`index.md` into pure projections.
3. Resolve `WaferEngine-staging`: flagged for the human to confirm merge-then-delete vs
   keep; not touched silently (data-loss risk).
4. Retire no-op status spam: collapse repeated "no new notes" lines into a single
   freshness timestamp going forward.
5. Update root `AGENTS.md` / `CLAUDE.md` / `HERMES.md` / `HUMAN.md` to describe the new
   source-vs-generated model and the write contract.

**In `claude-skills`:** add the `agent-memory/` skill package.

**In Hermes:** slim `synced-project-memory-vault` + the cron prompt to the one-line
delegation; move its 6 reference docs into the skill's `references/`.

### 6. Invocation & automation model

The system is automatic by default; manual invocation is only an escape hatch.

**Claude Code (remote, during coding) — two automatic layers:**

1. *Session start — write contract loads via always-read instructions.* The work
   repo's `CLAUDE.md`/`AGENTS.md` routing snippet points every session at `context.md`
   + the write-contract, so writes follow the format without being asked. Instruction-
   driven (reliable in that repo).
2. *Per-turn — deterministic regen via a `Stop` hook.* The `Stop` hook fires at the end
   of each assistant response turn (not on window close — Claude Code has no reliable
   close event, and this makes the design robust to closing/resuming a session). The
   hook first `git diff`s the project's memory paths and **exits instantly if nothing
   changed**, so ordinary coding turns are no-ops. When memory did change, it runs
   `build_views.py` + structure check (rebuild `timeline.md`/`index.md`, refresh
   freshness, flag dup candidates) and **commits locally, does not push**. Hooks are
   shell commands, so only the deterministic half runs here — at zero token cost. The
   push is deferred to a deliberate sync point (the agentic maintain pass or the human)
   to avoid unsupervised mid-session pushes and surprise binary-slide conflicts. Commit
   consolidation strategy (rolling/amended commit vs many-then-squash) is a Phase-1
   detail.

**Hermes (local) — already automatic:** the daily cron *is* the full agent-driven
maintain pass (drain, distill, synthesize prose views, mark conflicts, commit). This
redesign only slims its inline prompt to delegate to the shared `PLAYBOOK.md`.

**Division of labor:** deterministic regen is automatic per-Claude-session (hook);
LLM curation is automatic once/day (Hermes cron) and available on demand. This is
deliberate — hooks cannot run agent turns, so the daily cron is where judgment-requiring
curation lives.

**Manual fallback / forcing the skill.** The Claude Code skill is named **`agent-memory`**
(dir name = invocable name). Force it with `/agent-memory`, optionally with a mode/arg:
`/agent-memory maintain <project>` (curate one project), `/agent-memory maintain` (all),
`/agent-memory write` (write-to-spec now). The skill `description` also lets Claude
auto-select the procedure when the user asks maintenance-shaped questions.

## Sequencing

- **Phase 1 — build & install the skill.** Playbook, write-contract, templates,
  scripts, bootstrap; install on Mac Claude Code, gala2 Claude Code, and Hermes; update
  root instruction files and the routing snippet.
- **Phase 2 — dogfood migration.** Run the skill's own `maintain` pass to migrate the 4
  existing projects and perform the one-time cleanup, so the skill proves itself on real
  data. Resolve `WaferEngine-staging` with the human.

## Open items for the human

- `WaferEngine-staging`: confirm merge-then-delete vs keep.

## Risks / considerations for the plan

- **Binary slide co-edit conflicts.** `meetings/` decks (pptx) are co-edited by the human
  (Mac) and agents (possibly remote gala2) through git. Binary files do not auto-merge, so
  simultaneous edits on two machines can produce unresolvable conflicts. The plan should
  adopt a light convention (one deck = one working file, edit on one machine at a time,
  pull-before-edit / commit-after) rather than a heavy locking mechanism. Flag, not block.

## Success criteria

- No fact appears hand-authored in more than one file across a project.
- Human routinely edits only `capture.md` and `plan.md`.
- A fresh remote Claude session, given the routing snippet, writes only to
  `topics/<slug>.md` (in place) or dated `inbox/` files, in the standard format.
- `timeline.md` reflects every dated event chronologically.
- Editing `PLAYBOOK.md` and running `git pull` changes behavior on all three surfaces.
- The daily maintain pass produces zero no-op status lines.
