Mode-B on WSE-3 — where a round's time goes

Two timelines, very different scales. The first request pays a one-time KV bring-up (~3.8 s); every accepted step after that is a steady rewind round (~18 ms). All bars are host perf_counter wall-time (device run, rc=0). Left→right = time order; widths are proportional within each panel.

1 · First request — one-time KV bring-up

≈ 3.82 s total happens once per request (driver rounds r=0 + r=1)
The prefill pod computes the KV and hands it to the decode pod, which ingests it and runs the first draft window. None of this recurs.

≈ 18.1 ms repeats once per accepted step (measured n=41: mean 18.17 ms, std 0.16)
After bring-up, each round only re-arms the kernel to the accepted position and drafts a new window. No KV moves — just a ~1 KB band.

Reading these honestly. Every bar is a host perf_counter bracket, so there is no wafer/host clock-skew inside a number — but that also means host wall-time of a blocking op, not on-wire bytes. In particular egress lumps the prefill compute with the KV read-back (can't be split without on-wafer TSC), and r=1's full-KV H2D + 1st window is taken by difference. The r=0/r=1 rtts are driver-clock; the sub-stages are pod-clock — the same movement is not summed across both.
