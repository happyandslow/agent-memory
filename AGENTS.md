# AGENTS.md — Agent Memory Operating Instructions

This repository is Le Xu's private durable memory layer for work projects. It is designed for remote-first projects where development happens on SSH servers, while Obsidian on the Mac provides a local reading/planning interface under `10-work/<project>`.

## Prime directive

Maintain **curated, compact, project-scoped memory**. Do not dump raw transcripts or noisy logs. Make future fresh agent sessions easier by preserving decisions, current state, repo paths, commands, pitfalls, and next steps in predictable files.

## File model (v2 — 2026-07)

- Human source: `capture.md`, `plan.md`, `memory/project.md`, `memory/topics/*`.
- Agent captures: `memory/inbox/YYYY-MM-DD-*.md` (default write; see write-contract).
- Human dumps: `meetings/` (freeform, co-edited).
- Generated (never hand-edit): `memory/context.md`, `tracking/status.md`,
  `memory/timeline.md`, `index.md`.
- Maintenance procedure: the `agent-memory` skill (`PLAYBOOK.md`). Claude runs
  deterministic regen on the Stop hook; Hermes cron runs full curation daily.

## Default context loading

When working in this repo or in a work project that points here:

1. Read this file.
2. Read the project-specific `projects/<project>/memory/context.md` — a **generated**
   thin router (current focus + pointers), not a hand-authored packet.
3. Read `projects/<project>/memory/project.md`.
4. Read only relevant topic notes under `projects/<project>/memory/topics/`.
5. Read `projects/<project>/tracking/status.md` for generated current state.
6. Read `projects/<project>/plan.md` for human-maintained roadmap/progress narrative.
7. Skim `projects/<project>/capture.md` (human braindump) if it exists and looks
   recently touched — it may contain un-curated context newer than the generated views.

Do **not** scan all projects or all transcript indexes unless the user asks for archaeology or cross-project context.


## Dated-file convention

Every newly created durable note or artifact in this memory repo must be date-addressable so humans and agents can track chronology from filenames alone.

- For project `docs/` trees, every file must start with an ISO date prefix: `YYYY-MM-DD-<slug>.<ext>`. This applies to Markdown notes, scripts, slides, PDFs, exported data, and other generated artifacts. Date directories such as `docs/YYYY-MM-DD/` are allowed but do not replace the filename prefix.
- For quick captures and human notes, use `memory/inbox/YYYY-MM-DD-<short-topic>.md` unless a project-specific convention is stricter.
- For transcript pointers and event notes, use `memory/transcripts/YYYY-MM-DD/<YYYY-MM-DD-<short-topic>.md>` or another project-documented dated pattern.
- Stable entrypoint/control files are exempt from filename dating because agents need predictable paths: root `README.md`, `AGENTS.md`, `CLAUDE.md`, `HERMES.md`, `HUMAN.md`; project `index.md`, `plan.md`, `tracking/status.md`, `tracking/conflicts.md`; and `memory/README.md`, `memory/context.md`, `memory/project.md`, and directory `README.md` files. These files should still include dated status/history sections when they record time-sensitive facts.
- If an agent finds an undated non-exempt file during maintenance, rename it to the dated convention and update links. If the date is ambiguous, preserve the file and record a manual conflict instead of guessing.

## File ownership rules

- `projects/<project>/capture.md`: human-authored, append-only braindump (todos, meeting scraps, any format). Constantly touched by the human; agents should not curate it away without draining it into the right place first.
- `projects/<project>/plan.md`: human-maintained roadmap and durable progress narrative, including the canonical next-actions checklist. Agents may propose edits, but should avoid overwriting it mechanically.
- `projects/<project>/memory/project.md`: stable project facts, repo paths, conventions, known remotes, commands, a single ContextBase pointer.
- `projects/<project>/memory/topics/*.md`: evergreen, one-per-subsystem deep knowledge. Agents may **append** to an existing topic note in place (a dated `## Updates` subsection) but may **not** create new topic files during a session — new topic files come only from the maintain pass.
- `projects/<project>/memory/inbox/`: default write target for agent captures during a session, named `YYYY-MM-DD-<topic>.md`. Drained by the maintain pass into topics/plan/decisions; do not treat as a permanent home.
- `projects/<project>/meetings/`: freeform, co-edited human+agent dump for meeting slides/materials. Names are freeform (exempt from the dated-filename convention). Maintenance reads it to distill decisions/TODOs but never reorganizes it unprompted.
- `projects/<project>/memory/context.md`, `tracking/status.md`, `memory/timeline.md`, `index.md`: **generated views**, each carrying a `<!-- GENERATED by agent-memory skill: do not hand-edit -->` header. Rebuilt by the skill's `build_views.py` (Stop hook) or the full maintain pass (Hermes cron); never hand-authored by a session.
- `projects/<project>/memory/transcripts/`: indexes/events/pointers. Do not store bulk raw transcripts by default.
- `projects/<project>/memory/agents/`: machine/agent-specific setup notes.

## What to commit

Commit:

- durable decisions and rationale;
- current work state and next actions;
- repo paths, branches, build/test commands;
- experiment summaries and result pointers;
- meeting/handoff notes;
- compact transcript indexes and event records;
- generated `tracking/status.md` snapshots when useful.

Do not commit:

- secrets, tokens, passwords, SSH keys;
- raw credential files;
- active SQLite/database files;
- full raw Claude/Hermes transcript stores by default;
- dependency directories or caches;
- large logs unless explicitly requested and justified.

## Update discipline

At the end of meaningful work on a project, follow the write contract:

1. Write a dated capture to `memory/inbox/YYYY-MM-DD-<topic>.md` — this is the default,
   standard-format write for session outcomes. Do not hand-edit `memory/context.md`,
   `tracking/status.md`, `memory/timeline.md`, or `index.md`; those are generated.
2. If you are clearly refining an *existing* topic's evergreen facts, you may append a
   dated `## Updates` subsection to that `memory/topics/<slug>.md` in place. Do not
   create new topic files during a session.
3. Update `memory/project.md` only if stable paths/conventions/tools changed.
4. If there were important events, append a pointer/index entry under `memory/transcripts/`.
5. Deterministic regeneration of the generated views happens automatically via the
   Claude Code `Stop` hook (see `templates/claude-stop-hook.md`), which is git-diff-gated
   and commits locally without pushing. Full curation (draining `capture.md`/`inbox/`,
   rebuilding views, decluttering, conflict flagging) is the maintain pass, run by the
   daily Hermes cron or on demand via the `agent-memory` skill — see its `PLAYBOOK.md`.
6. If you ran the maintain pass yourself, commit with a concise message, e.g.
   `memory(project-x): update eval status and next steps`.

## Conflict handling

If git conflicts occur, preserve user-authored `plan.md` content first, then reconcile generated files. For generated `tracking/status.md`, regeneration is usually preferable to manual conflict surgery.

## Security/confidentiality

Assume work-project memory may contain sensitive research context. Keep this repository private. If a project has a stricter access boundary than the rest, move it to a separate memory repo.
